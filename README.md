## рҹӨ” What Is This?

**Backend Max** is a [Model Context Protocol (MCP)](https://modelcontextprotocol.io/) server that gives Claude deep diagnostic vision into your backend. It statically analyzes your codebase вҖ” both frontend *and* backend вҖ” to surface bugs, contract drift, missing validation, security gaps, and performance anti-patterns that no other single tool catches.

Think of it as a **senior backend engineer** that reviews every route, checks every contract, and never goes on vacation.

## рҹ”¬ Features Deep Dive

### рҹ”— Cross-Boundary Contract Verification

> **The #1 feature.** No other tool does this.

Backend Max reads your frontend fetch/axios calls *and* your backend route handlers, then cross-references them to find:

- **URL mismatches** вҖ” frontend calls `/api/user`, backend serves `/api/users`
- **Method mismatches** вҖ” frontend sends `POST`, backend expects `PUT`
- **Payload drift** вҖ” frontend sends `{ name, email }`, backend expects `{ username, email }`
- **Response shape drift** вҖ” frontend destructures `data.items`, backend returns `data.results`

```typescript
// Frontend: lib/api.ts
const user = await fetch('/api/user', {    // вҶҗ "/user" (singular)
  method: 'POST',
  body: JSON.stringify({ name: 'Alice' })  // вҶҗ sends "name"
});

// Backend: app/api/users/route.ts          // вҶҗ "/users" (plural)
export async function POST(req: Request) {
  const { username } = await req.json();    // вҶҗ expects "username"
  // ...
}
```

```
вҡ пёҸ  CONTRACT DRIFT DETECTED

   Route Mismatch:
   Frontend calls  вҶ’  /api/user   (POST)
   Backend serves  вҶ’  /api/users  (POST)

   Payload Mismatch:
   Frontend sends  вҶ’  { name: string }
   Backend expects вҶ’  { username: string }
```

### рҹ§  Intent-Aware Diagnosis

Backend Max doesn't just scan code вҖ” it **understands what you're building**. By analyzing your project structure, route names, and data models, it builds an intent map:

- Identifies your **domains** (auth, billing, users, etc.)
- Understands **relationships** between entities
- Flags issues that are contextually wrong, not just syntactically wrong

> рҹҺҜ *"This route handles payments but doesn't validate the amount field"* is more useful than *"missing validation on line 47"*.

---

### рҹӣЎпёҸ Pre-Deploy Safety Net

Six audit engines run in parallel, each targeting a category of bugs that commonly ship to production:

| # | Audit Engine | What It Catches |
|---|---|---|
| 1 | рҹ”— **API Contract Drift** | FrontendвҶ”backend URL, method, payload, and response mismatches |
| 2 | рҹҡЁ **Error Handling** | Missing try/catch, inconsistent error response formats, unhandled promise rejections |
| 3 | вң… **Input Validation** | Routes without Zod schemas, raw `req.body` access, missing type coercion |
| 4 | рҹ”‘ **Environment Variables** | Undefined refs, missing `.env` entries, `NEXT_PUBLIC_` prefix misuse |
| 5 | рҹ”’ **Security** | Auth middleware gaps, permissive CORS, SQL/NoSQL injection patterns, exposed secrets |
| 6 | вҡЎ **Performance** | N+1 queries, unbounded `SELECT *`, large payload serialization, missing pagination |

Each issue includes:
- **Severity** вҖ” `critical` В· `warning` В· `info`
- **Location** вҖ” exact file and line number
- **Explanation** вҖ” what's wrong and *why* it matters
- **Fix suggestion** вҖ” actionable code-level guidance

```markdown
|-------|--------|
```

The docs update every time you run a diagnosis. **They can never go stale.**

```
Found вҶ’ Acknowledged вҶ’ Fixed вҶ’ Verified вҶ’ (Regressed?)
```

## рҹӣЎпёҸ Safety & Security

Backend Max is built with a **safety-first philosophy**. Every operation вҖ” from file scanning to report generation вҖ” passes through multiple safety layers. Your code is never executed, your secrets are never stored, and all output is sanitized before it touches disk. Backend Max is designed so that even in the worst case, it cannot leak sensitive data or damage your project.

### Safety Systems

| System | What It Does | Default |
|--------|-------------|---------|
| **Path Guardian** | Validates project paths, blocks sensitive directories (.ssh, .aws, .gnupg), prevents path traversal attacks | Always on |
| **Output Sanitizer** | Detects and redacts secrets in reports вҖ” AWS keys, GitHub tokens, Stripe keys, JWTs, connection strings, private keys, Slack tokens, and 12+ more patterns | Always on |
| **Scope Limiter** | Caps file count (5,000), file size (1MB), scan depth (15 levels). Prevents memory exhaustion on large monorepos | Configurable |
| **Auto-Gitignore** | Automatically adds `.backend-doctor/` to `.gitignore` on first run. Prevents diagnostic data from being committed | On by default |
| **Report Pruning** | Auto-deletes diagnosis reports older than 30 days | Configurable |
| **Env Value Stripping** | Environment variable VALUES are never read or stored вҖ” only variable NAMES. Impossible to leak secrets through reports | Always on |
| **Write Protection** | Fix engine validates every write target: must be a source file, must be in project, must not be generated/config/lock files | Always on |

### Security Guarantees

- вң… Backend Max NEVER reads environment variable values вҖ” only names
- вң… Backend Max NEVER executes your code вҖ” pure static analysis
- вң… Backend Max NEVER sends data externally вҖ” everything stays local
- вң… All diagnostic output is scrubbed for 15+ secret patterns before writing to disk
- вң… `.backend-doctor/` is auto-gitignored to prevent accidental commits
- вң… File writes (fix engine) are sandboxed to source files within the project only

### Configuration

Safety limits can be tuned via `backend-max.config.json`:

```json
{
  "maxFiles": 5000,
  "maxFileSizeBytes": 1048576,
  "maxScanDepth": 15,
  "reportRetentionDays": 30,
  "autoGitignore": true
}
```
